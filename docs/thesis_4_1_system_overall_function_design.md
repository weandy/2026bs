# 4.1 系统总体功能设计

在完成社区卫生服务中心管理系统的需求分析后，可以进一步将系统的整体结构与功能模块进行归纳。结合本系统当前的实际实现情况，系统采用前后端分离的设计思想，前端负责界面展示与交互，后端负责业务处理、权限控制、数据访问与定时任务执行。为了便于论文撰写与后续系统说明，本文将该系统划分为七个层次，各层次之间职责清晰、边界明确，共同支撑居民端、医护端和管理员端三类业务场景。

## 4.1.1 系统分层架构设计

本系统采用分层设计，整体可划分为用户接入层、前端展示层、认证授权层、业务应用层、消息与调度层、数据访问层以及数据存储层七个层次，各层级及作用分别如下。

### 1. 用户接入层

用户接入层是系统面向外部用户的入口层，主要承载居民、医护人员、系统管理员等不同角色对系统的访问请求。居民用户主要通过浏览器访问居民端页面，完成预约挂号、档案查看、疫苗预约、签约随访等操作；医护人员通过医护端完成接诊、处方管理、档案维护、随访记录与接种登记等工作；管理员则通过后台管理端完成平台运维、人员管理、排班管理、公告管理和库存管理等操作。此外，系统还提供候诊公屏页面，用于在公共显示设备上展示候诊叫号信息，从而满足线下门诊场景的辅助展示需求。

### 2. 前端展示层

前端展示层主要负责界面渲染、交互组织、路由跳转和用户操作反馈。本系统前端基于 Vue 3 构建，配合 Element Plus、Pinia、Axios、Vue Router 等技术实现居民端、医护端和管理员端的统一前端框架。该层通过不同布局组件和路由配置区分三类角色的页面入口，使系统能够根据用户登录身份动态展示对应功能模块。居民端重点面向预约服务、健康档案、家庭成员和消息通知等业务；医护端重点面向接诊工作台、居民档案、签约随访、接种登记和处方记录等业务；管理员端则重点支持数据看板、人员管理、排班管理、公告管理、库存管理和系统审计等后台功能。

### 3. 认证授权层

认证授权层主要负责用户身份校验、登录认证、访问控制和权限拦截，是保障系统安全运行的重要组成部分。本系统后端使用 Spring Security 与 JWT 令牌机制实现统一认证，用户登录成功后由系统签发访问令牌，后续请求通过过滤器完成令牌解析和身份恢复。系统依据居民、医生、护士、管理员等角色进行权限划分，并在接口层面对 `/resident/**`、`/medical/**`、`/admin/**` 等路径实施差异化授权控制，从而保证不同角色只能访问其权限范围内的业务功能。

### 4. 业务应用层

业务应用层是系统的核心处理层，主要负责封装各类业务规则并向前端提供接口服务。该系统后端采用 Spring Boot 3 构建，并按照业务域拆分为多个 Maven 模块，包括公共模块、安全模块、居民模块、医护模块、管理员模块和启动模块。其中，居民模块主要负责预约挂号、就诊记录、健康档案、疫苗服务、家庭成员和消息通知等功能；医护模块主要负责接诊工作台、档案管理、随访管理、接种管理、处方处理和排班查看等功能；管理员模块主要负责数据看板、医护管理、居民管理、签约管理、公告管理、排班管理、科室管理、药品管理、疫苗库存和操作日志等功能。通过模块化设计，系统各业务域之间保持较好的独立性，也便于后续扩展和维护。

### 5. 消息与调度层

消息与调度层主要负责系统中的异步提醒、定时任务处理和状态维护。本系统已实现预约超时处理、随访提醒、消息推送、库存预警等定时业务逻辑，用于保证平台中的预约状态、随访计划、接种提醒和库存预警信息能够及时更新。该层能够降低人工维护成本，并增强系统在基层医疗服务场景中的连续运行能力。对于居民端消息通知、随访提醒、接种提醒等功能而言，消息与调度层为其提供了重要的后台支撑。

### 6. 数据访问层

数据访问层主要负责业务对象与数据库之间的数据交互。本系统采用 MyBatis-Plus 实现数据持久化，通过 Mapper 接口、分页插件和乐观锁机制完成增删改查、分页查询以及并发控制等操作。系统还通过双数据源配置对居民域数据和管理域数据进行区分访问，从而提升数据组织的清晰度与业务隔离性。数据访问层的存在，使上层业务逻辑无需直接操作底层数据库语句，增强了代码结构的规范性和可维护性。

### 7. 数据存储层

数据存储层主要负责系统核心业务数据的持久化保存。本系统目前采用 MySQL 数据库作为主要的数据存储载体，并按业务边界划分为 `chp_resident` 和 `chp_admin` 两个数据库，其中前者主要存储居民预约、就诊、档案、消息、疫苗预约等居民域数据，后者主要存储员工、角色、排班、公告、签约、药品库存、疫苗库存及系统日志等后台管理数据。此外，系统还结合 Caffeine 缓存机制对部分高频读取的数据进行本地缓存，以提高字典项、配置项和基础信息查询的访问效率。

## 4.1.2 系统功能结构设计

结合本系统当前前端已展示并投入使用的功能模块，可以将系统总体功能划分为居民服务模块、医护业务模块、综合管理模块以及公共服务与安全支撑模块四个部分。系统功能结构如图 4-2 所示。

图 4-2 系统功能结构图

### 1. 居民服务模块

居民服务模块主要面向社区居民，承担线上就医服务和个人健康信息查询等功能，是系统的重要服务入口。该模块当前主要包括首页、预约挂号、就诊记录、我的档案、疫苗接种、签约与随访、家庭成员、消息通知和个人中心等内容。其中，首页用于展示健康概览、快捷入口和近期业务提醒；预约挂号用于完成按科室、医生和日期的预约操作；就诊记录用于查看历次就诊、诊断信息和处方明细；我的档案用于查看血压、血糖、体重等体征趋势数据；疫苗接种用于管理接种预约和接种历史；签约与随访用于查看家庭医生签约状态和随访计划；家庭成员用于实现家属代管与代预约；消息通知用于查看系统消息、随访提醒和接种提醒；个人中心则用于维护居民个人资料和登录密码。

### 2. 医护业务模块

医护业务模块主要面向医生与护士，用于支撑社区卫生服务中心日常诊疗和健康管理业务。该模块当前主要包括接诊工作台、居民档案、处方记录、我的排班、工作统计、签约与随访、接种管理和个人中心等内容。其中，接诊工作台主要服务于医生的叫号接诊、诊断录入和处方开立；居民档案用于查询居民健康档案并维护过敏史、慢病标签等信息；处方记录用于查询历史处方及明细；我的排班用于查看个人班次并发起调班申请；工作统计用于统计本月接诊量、处方量及近期趋势；签约与随访用于对辖区签约居民进行管理并录入随访记录；接种管理主要用于护士登记疫苗接种情况和接种记录；个人中心则用于维护医护人员个人基础信息和账号资料。

### 3. 综合管理模块

综合管理模块主要面向系统管理员，用于完成平台的统一管理、资源配置和运行监督。该模块当前主要包括数据看板、医护管理、居民管理、签约管理、公告管理、排班管理、科室管理、药品管理、疫苗库存和操作日志等内容。其中，数据看板用于展示签约人数、随访计划、员工数量、公告数量、今日预约及就诊趋势等关键统计数据；医护管理用于维护医生、护士和管理员账号；居民管理用于查询居民信息并管理账号状态；签约管理用于查看和审核家庭医生签约业务；公告管理用于发布和下架系统公告，并向居民端推送通知；排班管理用于完成排班维护和换班审批；科室管理用于配置社区卫生服务中心的科室信息；药品管理用于进行药品库存查询、库存预警和出入库日志管理；疫苗库存用于维护疫苗库存、入出库记录及效期信息；操作日志用于记录后台关键操作行为，为系统审计和问题追踪提供依据。

### 4. 公共服务与安全支撑模块

公共服务与安全支撑模块主要负责为三端业务提供统一入口与基础支撑能力。该模块包括统一登录、候诊公屏、角色跳转、权限控制、消息推送和操作审计等功能。其中，统一登录用于根据用户身份进入对应业务端；候诊公屏用于展示科室叫号和候诊信息；权限控制依赖认证授权层实现不同角色的访问隔离；消息推送用于支撑居民端消息通知、随访提醒和公告推送；操作审计则用于记录后台关键业务操作，提升系统运行的可追溯性和安全性。

## 4.1.3 系统总体功能设计说明

从总体设计角度看，本系统围绕“居民服务、医护业务、后台管理”三个核心方向展开，通过前后端分离和分层架构的方式，构建了较为完整的社区卫生服务中心信息化平台。前端侧按照不同角色组织业务页面，保证界面展示和操作流程的针对性；后端侧则通过模块化结构封装业务逻辑，并结合权限控制、双数据源管理、消息调度与缓存机制，提高系统在基层医疗场景下的稳定性与扩展性。该总体功能设计既满足了居民线上服务需求，又支持医护人员日常诊疗和管理员后台运维，为后续数据库设计、详细模块设计和系统实现提供了清晰的结构基础。
